InkkOops

Recording real sessions and reading the artifacts

Recording is the fastest path from "I can reproduce this by hand" to "the runtime can replay it and capture proof." InkkOops records real window resizes and pointer activity, then turns that into a replayable automation script plus machine-readable diagnostics.

Record a session

dotnet run --project InkkSlinger.csproj -- --inkkoops-record

dotnet run --project InkkOops.Cli/InkkOops.Cli.csproj -- record --launch

Use the app normally. When you are done, close the app normally. The runtime writes a directory under artifacts/inkkoops-recordings by default.

End-to-end runtime examples

Example 1: start the app in recording mode

dotnet run --project InkkSlinger.csproj -- --inkkoops-record

Then interact with the app manually and close it normally. A new directory will appear under artifacts/inkkoops-recordings, for example:

artifacts/inkkoops-recordings/20260328-120250760-recorded-session/
using System.Diagnostics;

var startInfo = new ProcessStartInfo
{
    FileName = "dotnet",
    WorkingDirectory = @"C:\Users\Shadow\Desktop\InkkSlinger",
    UseShellExecute = false
};

startInfo.ArgumentList.Add("run");
startInfo.ArgumentList.Add("--project");
startInfo.ArgumentList.Add("InkkSlinger.csproj");
startInfo.ArgumentList.Add("--");
startInfo.ArgumentList.Add("--inkkoops-record");

using var process = Process.Start(startInfo)!;
process.WaitForExit();

Example 2: replay the exact recorded session through the app

dotnet run --project InkkSlinger.csproj -- --inkkoops-recording "C:\Users\Shadow\Desktop\InkkSlinger\artifacts\inkkoops-recordings\20260328-120250760-recorded-session\recording.json"

This launches the app, replays the recorded actions, waits for diagnostics stability, captures the final frame, dumps telemetry, writes artifacts, and exits automatically.

using System.Diagnostics;

var recordingPath =
    @"C:\Users\Shadow\Desktop\InkkSlinger\artifacts\inkkoops-recordings\20260328-120250760-recorded-session\recording.json";

var startInfo = new ProcessStartInfo
{
    FileName = "dotnet",
    WorkingDirectory = @"C:\Users\Shadow\Desktop\InkkSlinger",
    UseShellExecute = false
};

startInfo.ArgumentList.Add("run");
startInfo.ArgumentList.Add("--project");
startInfo.ArgumentList.Add("InkkSlinger.csproj");
startInfo.ArgumentList.Add("--");
startInfo.ArgumentList.Add("--inkkoops-recording");
startInfo.ArgumentList.Add(recordingPath);

using var process = Process.Start(startInfo)!;
process.WaitForExit();

Example 3: replay into a custom artifact directory

dotnet run --project InkkSlinger.csproj -- `
  --inkkoops-recording "C:\Users\Shadow\Desktop\InkkSlinger\artifacts\inkkoops-recordings\20260328-120250760-recorded-session\recording.json" `
  --inkkoops-artifacts "artifacts/inkkoops-replay-manual"

Use this when you want to keep multiple replay runs separate while iterating on a bug.

using System.Diagnostics;

var startInfo = new ProcessStartInfo
{
    FileName = "dotnet",
    WorkingDirectory = @"C:\Users\Shadow\Desktop\InkkSlinger",
    UseShellExecute = false
};

startInfo.ArgumentList.Add("run");
startInfo.ArgumentList.Add("--project");
startInfo.ArgumentList.Add("InkkSlinger.csproj");
startInfo.ArgumentList.Add("--");
startInfo.ArgumentList.Add("--inkkoops-recording");
startInfo.ArgumentList.Add(
    @"C:\Users\Shadow\Desktop\InkkSlinger\artifacts\inkkoops-recordings\20260328-120250760-recorded-session\recording.json");
startInfo.ArgumentList.Add("--inkkoops-artifacts");
startInfo.ArgumentList.Add("artifacts/inkkoops-replay-manual");

using var process = Process.Start(startInfo)!;
process.WaitForExit();

What a recording contains

  • recording.json: raw recorded actions such as resize, pointer move, pointer down, pointer up, wheel, and frame waits.
  • recorded-script.txt: a human-readable builder snippet that mirrors the session in InkkOopsScriptBuilder form.

The recording file is what the runtime replays. The text file is there so you can inspect or manually convert the session into a focused built-in repro script.

Replay a recorded session

dotnet run --project InkkSlinger.csproj -- --inkkoops-recording "C:\Users\Shadow\Desktop\InkkSlinger\artifacts\inkkoops-recordings\20260328-001157740-recorded-session\recording.json"

The replay loader appends a final diagnostics tail automatically:

  • WaitForIdle(InkkOopsIdlePolicy.DiagnosticsStable)
  • CaptureFrame("<recording-name>-final")
  • DumpTelemetry("<recording-name>-final")

What a run artifact contains

  • result.json: run status, command count, failure metadata, and total duration.
  • commands.log: time-stamped command descriptions in execution order.
  • command-###.json: one machine-readable diagnostics file per command.
  • *.png: frame captures written by CaptureFrame.
  • *.txt: telemetry dumps written by DumpTelemetry.

How to read the diagnostics

result.json

Check this first. It tells you whether the run completed, how many commands executed, which command failed, and the failure message if the run aborted.

commands.log

Use this to reconstruct the timeline quickly. It is the fastest way to verify that the runtime actually executed the steps you think it executed.

command-###.json

Each file captures the execution mode, selector, resolution result, matched element, matched peer, bounds, viewport, anchor, action point, hover/focus before and after, and failure category if something went wrong.

Frame and telemetry files

The final .png answers what the user actually saw. The matching .txt answers what the framework thought the tree looked like at that moment.

Failure categories you will see

  • Unresolved
  • Ambiguous
  • Unrealized
  • Offscreen
  • Clipped
  • Disabled
  • NotInteractive
  • SemanticProviderMissing
  • UiThreadViolation
  • Timeout

Practical debugging loop

  1. Record the real repro.
  2. Replay it and inspect the final frame.
  3. Open result.json and commands.log to confirm the replay executed correctly.
  4. Open the relevant command-###.json files to inspect selector resolution and actionability decisions.
  5. Promote the useful part of the replay into a built-in script once the sequence is stable.
  6. Add a regression test once the actual subsystem bug is isolated.
