ProgressBar

ProgressBar is the framework's non-interactive range control for showing determinate or indeterminate work. It shares the same range coercion foundation as Slider and ScrollBar, but its shipped template is driven by named parts plus template-root visual states rather than direct user manipulation.

Quick start

Use ProgressBar when your UI needs to display progress, load state, or background activity without exposing direct input. The control can render as a classic determinate fill, a vertical meter, or an indeterminate animation when there is no stable percentage yet.

Minimal determinate bar

<ProgressBar Width="260"
             Minimum="0"
             Maximum="100"
             Value="64" />

Vertical progress

<ProgressBar Width="8"
             Height="140"
             Orientation="Vertical"
             Minimum="0"
             Maximum="1"
             Value="0.3" />

Indeterminate progress

<ProgressBar Width="220"
             Height="10"
             IsIndeterminate="True" />

Choose the right range control: use ProgressBar when the user should observe progress only. Use Slider when the user must choose a value, and use ScrollBar when the range represents viewport movement through content.

Range model

ProgressBar inherits from RangeBase, so Minimum, Maximum, and Value obey the same coercion rules as the other range controls. That shared contract drives the normalized fill geometry, automation exposure, and parity tests for out-of-range assignments.

Bounds and coercion

Minimum defaults to 0, Maximum defaults to 100, and Value is clamped back into the effective range every time any of those inputs changes.

Non-interactive by design

The control is not hit-testable by default. It reports state, but unlike Slider it does not expose drag or keyboard interaction paths for changing the value.

Horizontal fill

In the default horizontal template, the indicator grows from left to right as the normalized value increases.

Vertical fill

In vertical orientation, the same normalized value is mapped from the bottom upward so lower values stay near the base of the bar.

var progressBar = new ProgressBar
{
    Width = 240f,
    Minimum = 0f,
    Maximum = 1f,
    Value = 0.42f
};

Template contract and visual states

The shipped theme uses a template-root VisualStateManager.VisualStateGroups declaration plus three named parts. When those parts exist, ProgressBar sizes and positions them directly during arrange and frame updates. When the template is incomplete, the control falls back to its direct renderer instead of throwing.

PART_Track

The track is the layout host for the indicator and glow. It provides the geometry the control uses for both determinate and indeterminate calculations.

PART_Indicator

The indicator is the filled segment. In determinate mode its width or height is mapped from the normalized value, and in indeterminate mode it becomes the moving chunk.

PART_GlowRect

The glow rectangle is only visible in the indeterminate state. The control updates its position each frame while the visual-state setter controls its opacity.

State groups

CommonStates switches between Determinate and Indeterminate. ValidationStates switches between Valid, InvalidFocused, and InvalidUnfocused.

<ControlTemplate TargetType="{x:Type ProgressBar}">
  <Border x:Name="RootChrome">
    <VisualStateManager.VisualStateGroups>
      <VisualStateGroup x:Name="CommonStates">
        <VisualState x:Name="Determinate" />
        <VisualState x:Name="Indeterminate">
          <VisualState.Setters>
            <Setter TargetName="PART_GlowRect" Property="Opacity" Value="0.8" />
          </VisualState.Setters>
        </VisualState>
      </VisualStateGroup>
    </VisualStateManager.VisualStateGroups>
    <Grid x:Name="PART_Track">
      <Border x:Name="PART_Indicator" />
      <Border x:Name="PART_GlowRect" Opacity="0" />
    </Grid>
  </Border>
</ControlTemplate>

Template visual states are resolved against the applied template root, so state setters and storyboards can target named template parts directly. State transitions also clear values from the previous state when the next state no longer owns that property.

Indeterminate mode and validation

Indeterminate mode does not estimate completion. Instead, the control advances a phase value each frame and uses that value to reposition the indicator and glow inside the track. Validation styling is independent of that animation and is controlled through the template's validation states.

Motion model

The default template keeps the glow visible only while IsIndeterminate is true. When indeterminate mode turns off, the phase resets and the glow opacity returns to zero.

Validation chrome

Validation.HasError and focus state drive the validation group so a templated bar can surface focused and unfocused invalid styling without changing the range model.

<ProgressBar Width="240"
             Height="12"
             IsIndeterminate="True"
             Foreground="{StaticResource OrangePrimaryBrush}"
             Background="{StaticResource DarkElevatedBrush}" />

Automation and regression coverage

ProgressBar exposes the read-only range value automation pattern expected for a passive progress indicator. The regression suite also verifies template visual-state resolution, horizontal and vertical indicator geometry, indeterminate motion, validation-state switching, fallback rendering for incomplete templates, and shared range coercion.

Automation

Automation peers report the current value through the range-value pattern, but the pattern is read-only because the control is not user-editable.

Key tests

Look at ProgressBarBehaviorTests, VisualStateManagerTemplateTests, AutomationPatternValueTests, and RangeBaseCoercionTests when changing ProgressBar behavior.
