InkkOops

Authoring built-in scripts

InkkOops v1 scripts are authored as normal C# classes in the app project. They are compiled into the assembly, discovered by reflection, and executed by name through the runtime service.

The contract

Implement IInkkOopsBuiltinScript. The runtime uses Name as the public entry point and CreateScript() to produce the command list.

using InkkSlinger;

public sealed class InkkOopsExampleScript : IInkkOopsBuiltinScript
{
    public string Name => "example-navigation";

    public InkkOopsScript CreateScript()
    {
        return new InkkOopsScriptBuilder(Name)
            .WaitForIdle(InkkOopsIdlePolicy.DiagnosticsStable)
            .Click("Button")
            .WaitForVisible("PreviewHost")
            .CaptureFrame("after-button-click")
            .DumpTelemetry("after-button-click")
            .Build();
    }
}

Selectors

Prefer selectors over coordinates whenever you can. They survive layout changes, templates, and viewport differences much better.

var byName = InkkOopsTargetSelector.Name("CatalogSidebarScrollViewer");
var byAutomationId = InkkOopsTargetSelector.AutomationId("PreviewHost");
var byAutomationName = InkkOopsTargetSelector.AutomationName("RichTextBox");

var richTextBoxInSidebar =
    InkkOopsTargetSelector.Within(
        InkkOopsTargetSelector.Name("CatalogSidebarScrollViewer"),
        InkkOopsTargetSelector.AutomationName("RichTextBox"));

var secondDeleteButton =
    InkkOopsTargetSelector.DescendantOf(
        InkkOopsTargetSelector.Name("ToolbarHost"),
        InkkOopsTargetSelector.AutomationName("Delete"))
    .WithIndex(1);

Resolution is strict. If no match exists, the command fails as Unresolved. If more than one match exists, the command fails as Ambiguous.

Waits and actionability

  • WaitForElement: the target resolves, nothing more.
  • WaitForVisible: the target exists and reports visible.
  • WaitForEnabled: the target exists and is enabled.
  • WaitForInViewport: the target exists and is within the viewport.
  • WaitForInteractive: the target exists, is visible, enabled, has usable bounds, and the chosen action point is hit-testable.
var richTextBoxButton = InkkOopsTargetSelector.Within(
    InkkOopsTargetSelector.Name("CatalogSidebarScrollViewer"),
    InkkOopsTargetSelector.AutomationName("RichTextBox"));

return new InkkOopsScriptBuilder("sidebar-richtextbox")
    .WaitForInteractive(richTextBoxButton)
    .Hover(richTextBoxButton)
    .Click(richTextBoxButton)
    .Build();

Builder surface at a glance

The current builder supports these command families:

  • Timing and layout: ResizeWindow, WaitFrames, WaitForIdle
  • Wait conditions: WaitForElement, WaitForVisible, WaitForEnabled, WaitForInViewport, WaitForInteractive
  • Pointer input: Hover, Click, MovePointer, PointerDown, PointerUp, Drag, Wheel
  • Semantic actions: Invoke, Activate, ScrollTo, ScrollBy, ScrollIntoView, ScrollIntoViewItemText, ScrollIntoViewItemIndex
  • Proof and assertions: CaptureFrame, DumpTelemetry, AssertExists, AssertNotExists, AssertProperty, AssertAutomationEvent

Keep scripts short and explicit. Prefer a small number of purposeful steps over long imperative recordings unless the bug genuinely depends on a long recorded path.

Pointer actions and anchors

Pointer actions exercise the real input path. Use them for hover bugs, hit-testing bugs, pointer capture bugs, drag operations, or any scenario where semantic automation would hide the problem.

new InkkOopsScriptBuilder("pointer-anchor-demo")
    .Hover("Normal", InkkOopsPointerAnchor.Center)
    .Click(
        InkkOopsTargetSelector.Name("ScrollThumb"),
        InkkOopsPointerAnchor.TopLeft)
    .Drag(
        InkkOopsTargetSelector.Name("ScrollThumb"),
        0f,
        120f,
        InkkOopsPointerAnchor.OffsetBy(4f, 4f))
    .Build();

Default anchor is Center. The runtime writes the final chosen root-space action point into the command diagnostics file.

Semantic actions and scrolling

Semantic commands route through automation peers and providers. They are appropriate when you want a stable control-level action and you do not need to test pointer behavior itself.

var sidebar = InkkOopsTargetSelector.Name("CatalogSidebarScrollViewer");
var richTextBoxButton = InkkOopsTargetSelector.Within(
    sidebar,
    InkkOopsTargetSelector.AutomationName("RichTextBox"));

return new InkkOopsScriptBuilder("semantic-sidebar-nav")
    .ScrollIntoView(sidebar, InkkOopsScrollLocator.ForElement(richTextBoxButton))
    .WaitForInteractive(richTextBoxButton)
    .Activate(richTextBoxButton)
    .AssertExists("PreviewHost")
    .Build();
return new InkkOopsScriptBuilder("items-control-scroll")
    .ScrollIntoViewItemText(
        InkkOopsTargetSelector.Name("CatalogSidebarScrollViewer"),
        "RichTextBox")
    .ScrollIntoViewItemIndex(
        InkkOopsTargetSelector.Name("ResultsList"),
        25)
    .Build();

Rule of thumb. If the bug is visual, input-sensitive, or hover-sensitive, use pointer commands. If the goal is navigation or state mutation and pointer fidelity does not matter, semantic commands are usually simpler.

Assertions and proof

  • AssertExists
  • AssertNotExists
  • AssertProperty
  • AssertAutomationEvent
  • CaptureFrame and DumpTelemetry for visual and structural proof
return new InkkOopsScriptBuilder("prove-selection")
    .Click("DataGrid")
    .WaitForVisible("PreviewHost")
    .AssertProperty("SelectionHeader", "Text", "Selected: DataGrid")
    .CaptureFrame("datagrid-selected")
    .DumpTelemetry("datagrid-selected")
    .Build();
