Controls

ScrollBar

ScrollBar is a direct-drawn range control that visualizes a viewport moving through a larger extent. It renders a track and a thumb, supports horizontal and vertical orientations, and is used internally by ScrollViewer for end-user scrolling chrome.

Quick start

Use ScrollBar when you want to expose scroll state, viewport position, or a range indicator on a custom surface. For standard content scrolling, you will usually use ScrollViewer, which owns and configures its internal scroll bars automatically.

Minimal horizontal bar

<ScrollBar Width="280"
           Orientation="Horizontal"
           Minimum="0"
           Maximum="100"
           ViewportSize="20"
           Value="35" />

Vertical bar for a tall surface

<ScrollBar Height="220"
           Orientation="Vertical"
           Minimum="0"
           Maximum="1200"
           ViewportSize="240"
           Value="360" />

Styled bar using control colors

<ScrollBar Width="300"
           Orientation="Horizontal"
           Minimum="0"
           Maximum="800"
           ViewportSize="160"
           Value="240"
           Background="#152231"
           Foreground="#6FB7FF" />

Default recommendation: reach for ScrollViewer when you need actual content scrolling. Use a standalone ScrollBar for custom editors, minimaps, timeline rails, camera indicators, or other surfaces where you own the range logic yourself.

What the range means

ScrollBar models a visible window inside a larger logical extent. The important point is that Maximum represents the end of the full extent, not the largest legal scroll offset.

Extent

The total logical length is Maximum - Minimum. This is the full content size or domain size the bar represents.

Viewport

ViewportSize is the visible window inside that extent. The thumb size is derived from it.

Offset

Value is the current viewport origin. For a typical scroller, the legal offset range is Minimum through Minimum + max(0, Maximum - Minimum - ViewportSize).

Clamping

Built-in drag and page-click logic clamps to the scrollable range. If you assign Value directly, keep it in range yourself for the clearest semantics.

<ScrollBar Width="320"
           Orientation="Horizontal"
           Minimum="0"
           Maximum="1000"
           ViewportSize="320"
           Value="180" />

In that example, the represented extent is 1000 units, the visible window is 320 units, and the largest practical offset is 680, not 1000.

Thumb sizing and movement

The thumb rectangle is computed directly from the track size, orientation, extent, viewport, and current value. The math is simple and predictable.

When the thumb fills the track

If Maximum - Minimum is effectively zero, the thumb covers the full track. This is how the control represents a non-scrollable extent.

Viewport-based ratio

When ViewportSize is positive, thumb length is based on ViewportSize / extent, clamped between 5% and 100% of the track.

Minimum thumb size

The rendered thumb never shrinks below 14 layout units, even when the represented extent is much larger than the viewport.

  • If ViewportSize is 0 or negative, the thumb falls back to roughly 10% of the track, still respecting the 14-unit minimum.
  • The thumb position is normalized across the scrollable range, not the full extent.
  • When the viewport covers the whole extent, the thumb stays effectively pinned to the start because there is no remaining travel.
<StackPanel>
  <ScrollBar Width="260"
             Orientation="Horizontal"
             Minimum="0"
             Maximum="100"
             ViewportSize="80"
             Value="10" />

  <ScrollBar Width="260"
             Orientation="Horizontal"
             Minimum="0"
             Maximum="100"
             ViewportSize="10"
             Value="10"
             Margin="0,8,0,0" />
</StackPanel>

The first bar shows a large thumb because the viewport covers most of the extent. The second shows a much smaller thumb because the viewport is only a small slice of the same range.

Layout and sizing

ScrollBar is not a content host. Its desired size is driven entirely by orientation and thickness.

Vertical orientation

The desired width is max(8, Thickness). The desired height follows the available height passed in by layout.

Horizontal orientation

The desired height is max(8, Thickness). The desired width follows the available width passed in by layout.

  • Thickness defaults to 12.
  • The cross-axis thickness has a practical minimum of 8, even if you set a smaller value.
  • Because the along-axis desired size mirrors available space, standalone bars usually work best when you also set an explicit Width or Height.
<Grid>
  <Grid.ColumnDefinitions>
    <ColumnDefinition Width="*" />
    <ColumnDefinition Width="Auto" />
  </Grid.ColumnDefinitions>

  <Canvas Grid.Column="0" Width="420" Height="220" />

  <ScrollBar Grid.Column="1"
             Height="220"
             Orientation="Vertical"
             Thickness="14"
             Minimum="0"
             Maximum="600"
             ViewportSize="220"
             Value="120" />
</Grid>

Interaction model

The built-in pointer behavior supports thumb dragging and paging by clicking the track outside the thumb. That behavior is straightforward, but there is an ownership detail that matters for end-user apps.

Thumb drag

Pressing on the thumb starts a drag. The control records the pointer offset inside the thumb so the thumb does not jump when dragging begins.

Track click

Clicking before or after the thumb moves the value by LargeChange toward the clicked side. The step is always at least 1.

  • LargeChange defaults to 32.
  • SmallChange defaults to 16, but the current built-in pointer path does not use it.
  • Drag math and track clicks clamp the resulting value to the legal scrollable range.

Important ownership rule: the framework's automatic mouse handling for ScrollBar is currently wired through an owning ScrollViewer. A standalone ScrollBar renders correctly and exposes range state, but direct end-user dragging and track paging are not automatically hooked up unless the bar is the internal bar owned by a ScrollViewer or you provide your own input logic.

Appearance and color fallback

ScrollBar draws two filled rectangles: the full track first, then the thumb on top. It does not use a control template or separate child elements for these visuals.

Track color

If TrackBrush is set, it wins. Otherwise, if Background is set on the control, the track uses Background. If neither is set, the default track color is a dark gray.

Thumb color

If ThumbBrush is set, it wins. Otherwise, if Foreground is set on the control, the thumb uses Foreground. If neither is set, the default thumb color is a lighter gray.

  • The track fills the full arranged slot.
  • The thumb is rendered as an axis-aligned rectangle.
  • The element's normal Opacity applies to both rectangles.
  • No rounded geometry, border stroke, or separate chrome layers are drawn by the control itself.
<StackPanel>
  <ScrollBar Width="280"
             Orientation="Horizontal"
             Minimum="0"
             Maximum="300"
             ViewportSize="80"
             Value="60"
             Background="#132131"
             Foreground="#77D1FF" />

  <ScrollBar Width="280"
             Orientation="Horizontal"
             Minimum="0"
             Maximum="300"
             ViewportSize="80"
             Value="60"
             TrackBrush="#1B2D41"
             ThumbBrush="#FFB347"
             Margin="0,8,0,0" />
</StackPanel>

The first bar uses the inherited control color fallback path. The second overrides the dedicated scroll-bar colors directly.

ScrollViewer ownership

ScrollViewer creates and owns its internal horizontal and vertical ScrollBar instances. When that happens, the viewer drives the important range properties for you.

Values supplied by ScrollViewer

The viewer sets Minimum to 0, Maximum to the content extent, ViewportSize to the current viewport size, and Value to the current scroll offset.

Thickness resolution

The viewer uses its own ScrollBarThickness by default, but an internally owned bar can also be influenced by the bar's explicit cross-axis size or Thickness.

<ScrollViewer Width="360"
              Height="220"
              HorizontalScrollBarVisibility="Auto"
              VerticalScrollBarVisibility="Auto"
              ScrollBarThickness="16">
  <Canvas Width="960" Height="640">
    <Border Canvas.Left="40"
            Canvas.Top="32"
            Width="200"
            Height="120"
            Background="#183149"
            BorderBrush="#5A9CD4"
            BorderThickness="1" />
  </Canvas>
</ScrollViewer>

If your goal is end-user scrolling of a content surface, this is the usual pattern. The viewer handles range updates, offset clamping, and pointer routing to its internal bars.

Property reference

Range properties

Orientation: Vertical or Horizontal. Default: Vertical.

Minimum: lower bound of the represented extent. Default: 0.

Maximum: upper bound of the represented extent. Default: 0.

Value: current viewport origin or logical offset. Default: 0.

ViewportSize: visible window size inside the extent. Default: 0.

Interaction and rendering

SmallChange: small step value exposed on the API. Default: 16. Not currently used by the built-in pointer path.

LargeChange: page step used for track clicks. Default: 32.

Thickness: desired cross-axis thickness. Default: 12. Practical minimum: 8.

TrackBrush: track fill color. Default: (42, 42, 42).

ThumbBrush: thumb fill color. Default: (112, 112, 112).

Inherited control colors

Background: fallback color source for the track when TrackBrush is still default.

Foreground: fallback color source for the thumb when ThumbBrush is still default.

Opacity: inherited rendering opacity applied to the track and thumb.

BorderBrush, BorderThickness, and Padding: inherited on the base control surface, but not used by the current ScrollBar layout or render path.

Automation

ScrollBar participates in the framework automation surface as a scroll-bar control type and exposes range values through the automation peer.

Patterns and examples

Viewport indicator for a large canvas

<Grid>
  <Grid.ColumnDefinitions>
    <ColumnDefinition Width="*" />
    <ColumnDefinition Width="Auto" />
  </Grid.ColumnDefinitions>

  <Canvas x:Name="WorldCanvas" Grid.Column="0" Width="1200" Height="900" />

  <ScrollBar Grid.Column="1"
             Height="260"
             Orientation="Vertical"
             Minimum="0"
             Maximum="900"
             ViewportSize="260"
             Value="{Binding CameraY}"
             TrackBrush="#13202E"
             ThumbBrush="#5FA8E8" />
</Grid>

Horizontal timeline navigator

<ScrollBar Width="420"
           Orientation="Horizontal"
           Minimum="0"
           Maximum="{Binding TimelineLength}"
           ViewportSize="{Binding VisibleRange}"
           Value="{Binding TimelineOffset}"
           Background="#0F1823"
           Foreground="#F5B04C" />

Programmatic setup in code

var bar = new ScrollBar
{
    Orientation = Orientation.Vertical,
    Height = 240f,
    Minimum = 0f,
    Maximum = documentHeight,
    ViewportSize = viewportHeight,
    Value = currentOffset,
    TrackBrush = new Color(19, 32, 46),
    ThumbBrush = new Color(95, 168, 232)
};

bar.Value = MathF.Max(
    bar.Minimum,
    MathF.Min(
        bar.Minimum + MathF.Max(0f, (bar.Maximum - bar.Minimum) - bar.ViewportSize),
        nextOffset));

Prefer ScrollViewer for interactive content scrolling

<ScrollViewer Width="340"
              Height="220"
              VerticalScrollBarVisibility="Auto">
  <StackPanel>
    <TextBlock Text="Entry 1" />
    <TextBlock Text="Entry 2" />
    <TextBlock Text="Entry 3" />
    <TextBlock Text="Entry 4" />
    <TextBlock Text="Entry 5" />
    <TextBlock Text="Entry 6" />
    <TextBlock Text="Entry 7" />
  </StackPanel>
</ScrollViewer>

Notes and pitfalls

Maximum is not the largest offset. The effective end of the scrollable range is Maximum - ViewportSize relative to Minimum, clamped at zero.

Standalone bars are not fully wired for end-user mouse input by default. Automatic drag and track-click routing currently comes from the owning ScrollViewer input path.

SmallChange is currently unused by the built-in pointer path. Do not rely on it for user interaction behavior unless you add your own logic around the control.

Background, Foreground, TrackBrush, and ThumbBrush are colors, not brush objects. Examples and styles for this control should use values that resolve to Color.

No template-driven parts. The control renders its visuals directly; it does not expose a separate thumb child, track child, or content slot.

Set an explicit length when using the control on its own. Because the desired size along the scrolling axis mirrors the available size, a standalone bar is easiest to reason about when you also set Width or Height.
