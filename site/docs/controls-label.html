Controls

Label

Label is a real WPF-style caption control. It derives from ContentControl, hosts arbitrary content, can point at a target control through Target, participates in access-key routing, and renders through a control template instead of behaving like a text leaf.

Quick start

Use Label for captions that describe another control, for access-key driven field labels, and for content that should benefit from chrome such as padding, border, and background through the default template.

Minimal label

<Label Content="Display name" />

Text-node content

<Label>Display name</Label>

Field label with mnemonic

<StackPanel>
  <Label Target="UserNameBox">_User name</Label>
  <TextBox x:Name="UserNameBox"
           Width="240" />
</StackPanel>

Default mental model: use Label when the text is a caption with semantics. Use TextBlock when you only need raw text rendering.

What Label is

This implementation now follows the WPF contract much more closely. Label is a templated content control, not a specialized TextBlock.

What it is

A ContentControl that can host strings, visuals, or templated content, and can associate itself with another control through Target.

What it is not

It is not a drop-in replacement for TextBlock. It no longer exposes a primary Text API or a text-leaf measure and render pipeline.

  • Content is the primary surface for strings, visuals, and bound objects.
  • Target points to the control that the label describes.
  • The default template applies Padding, Background, BorderBrush, BorderThickness, and content alignment.
  • Label is non-focusable by default. Access keys move focus to the target instead of to the label.

Content model

Label behaves like a normal single-content control. That gives you three common authoring patterns.

String content

Assign a string through Content or use a text node in XAML. When access-key recognition is enabled in the template, plain strings are presented through AccessText.

Visual content

You can host a child element directly, such as a TextBlock, AccessText, or a small icon-plus-text panel.

  • Use string content for the common label case.
  • Use child visuals when you need richer composition than a single string.
  • Use ContentTemplate when the content is data and you want a consistent label presentation shape.
  • Whitespace-only text nodes are ignored by the XAML loader.

Label with explicit visual content

<Label Target="SearchBox"
       Padding="8,4"
       Background="#122233"
       BorderBrush="#31506B"
       BorderThickness="1">
  <StackPanel Orientation="Horizontal">
    <TextBlock Text="Search" />
    <TextBlock Text=" (optional)"
               Foreground="#89A9C3" />
  </StackPanel>
</Label>

Label with a content template

<Label Content="{Binding ActiveProfile}"
       Padding="8">
  <Label.ContentTemplate>
    <DataTemplate>
      <StackPanel Orientation="Horizontal">
        <TextBlock Text="Profile: " FontWeight="Bold" />
        <TextBlock Text="{Binding}" />
      </StackPanel>
    </DataTemplate>
  </Label.ContentTemplate>
</Label>

Target and access keys

Label.Target gives the control real caption semantics. When the label contains a mnemonic such as _Name, the access-key pipeline can move focus to the target control.

Target resolution

You can assign Target with a direct object reference, with {x:Reference ...}, or with a plain element name such as Target="UserNameBox".

Automation

Assigning Target also updates AutomationProperties.LabeledBy on the target, which improves accessibility and automation lookup behavior.

Named target

<StackPanel>
  <Label Target="EmailBox">_Email</Label>
  <TextBox x:Name="EmailBox"
           Width="260" />
</StackPanel>

x:Reference target

<StackPanel>
  <TextBox x:Name="PasswordBox"
           Width="260" />
  <Label Target="{x:Reference PasswordBox}">_Password</Label>
</StackPanel>

Access-key rule: the mnemonic is recognized from the label's presented content. If the label is just decorative text, use TextBlock instead of adding stray access-key semantics.

Layout, template, and styling

The default label template is a bordered content host. Unlike the old implementation, chrome and inner spacing are real parts of the control surface.

Chrome

Background, BorderBrush, and BorderThickness apply through the default template.

Inner spacing

Padding changes both desired size and the content placement inside the label.

Alignment

HorizontalContentAlignment and VerticalContentAlignment control how hosted content is arranged inside the template.

Styled label surface

<Label Content="Connection"
       Padding="10,6"
       Background="#112031"
       BorderBrush="#2F4C66"
       BorderThickness="1"
       Foreground="#E4F1FF"
       FontWeight="SemiBold" />

Centered content inside the label

<Label Width="220"
       Height="56"
       Content="Ready"
       Padding="8"
       HorizontalContentAlignment="Center"
       VerticalContentAlignment="Center"
       Background="#122638"
       BorderBrush="#33526C"
       BorderThickness="1" />

Style targeted at labels

<Style x:Key="FieldLabelStyle" TargetType="{x:Type Label}">
  <Setter Property="Foreground" Value="#C8D9EA" />
  <Setter Property="FontSize" Value="14" />
  <Setter Property="FontWeight" Value="SemiBold" />
  <Setter Property="Margin" Value="0,0,0,4" />
  <Setter Property="Padding" Value="6,2" />
</Style>

Where labels appear automatically

The framework still creates Label instances in some presentation fallbacks. Those generated labels now participate in the same real control pipeline as hand-authored labels.

ContentPresenter fallback

Plain content with no template can be materialized as a generated label, with fonts and foreground copied from the owning control.

Header fallback

HeaderedItemsControl can generate a label for plain-data headers when no header template is supplied.

Access-key aware text

When a presenter recognizes access keys, generated text content is surfaced through AccessText instead of a raw text leaf.

<ContentControl Content="Build succeeded" />

That content may end up presented by a generated Label if no explicit template or visual child is supplied.

Notes and pitfalls

No primary Text property. Migrate old code to Content or to a child TextBlock when you want explicit text rendering behavior.

Use TextBlock for raw text. Calendar headers, status text, dense item visuals, and text-heavy rendering surfaces should usually stay on TextBlock.

Do not mix implicit text and child visuals. A single-content control can have text-node content or an implicit child, not both at the same level.

Target matters for semantics. If the label describes an input, wire the target so focus and automation behavior stay correct.
