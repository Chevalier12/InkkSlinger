InkkOops

Runtime UI automation for the framework itself

InkkOops is the framework's in-process automation and diagnostics runtime. It exists to make hard UI bugs reproducible: resize sequences, hover transitions, scrolling edge cases, layout regressions, retained-render issues, and parity investigations that only show up in the live app.

It is not an external accessibility driver. It runs inside the app, targets the real UiRoot, and executes either semantic automation actions or synthetic pointer/input actions against the same tree the runtime is rendering.

Use it for both automated runtime verification and manual testing workflows. The same system can drive scripted repros, record a human session, replay that session later, and capture the proof artifacts needed to debug what happened.

What it gives you

Deterministic repro scripts

Built-in scripts are compiled into the main app assembly and run against the real runtime. You can launch them at startup, inject them into a running app, or replay a recorded user session.

Proof artifacts

Every run writes machine-readable output such as result.json, commands.log, command-###.json, frame captures, and telemetry dumps.

Pointer and semantic actions

Use pointer actions when you need real hit-testing, hover, drag, and scroll behavior. Use semantic actions when you need stable control-level intent through automation peers.

Recorded sessions

You can record the actual steps a human took in the app, including window resizes and pointer activity, then replay that exact session later with capture and diagnostics enabled.

Core concepts

Built-in scripts

A built-in script is a class that implements IInkkOopsBuiltinScript. The runtime discovers these by reflection in the app assembly and exposes them by name for launch, attach, and replay workflows.

Selectors

Targets are resolved through InkkOopsTargetSelector, not fragile coordinate guesses. Resolution prefers explicit x:Name, then automation identifiers and automation names where appropriate, and it reports ambiguity instead of silently picking the first match.

Actionability

WaitForElement only means the element resolves. If you need a target that can actually be clicked or hovered, use WaitForVisible, WaitForInViewport, or WaitForInteractive.

Artifacts

Runs are written under artifacts/inkkoops by default. Recordings are written under artifacts/inkkoops-recordings by default.

Quick start

List the scripts that are compiled into the app:

dotnet run --project InkkOops.Cli/InkkOops.Cli.csproj -- list

Launch the app and run a built-in script immediately:

dotnet run --project InkkSlinger.csproj -- --inkkoops-script "sidebar-button-richtextbox"

Record a manual session:

dotnet run --project InkkSlinger.csproj -- --inkkoops-record

Replay a recording and capture the final frame:

dotnet run --project InkkSlinger.csproj -- --inkkoops-recording "C:\path\to\recording.json"

Built-in scripts currently in the repo

  • buttons-resize-chaos-repro: aggressive button-view resize and hover repro used for render and layout debugging.
  • buttons-resize-hover-repro: narrower button-view repro focused on resize plus hover transitions.
  • datagrid-workbench-capture: navigates to the DataGrid validation surface and captures proof artifacts.
  • sidebar-button-richtextbox: drives the controls catalog sidebar to the RichTextBox view and verifies navigation.

Built-in only in v1. InkkOops does not compile arbitrary external scripts at runtime. Scripts are authored as normal C# classes in the app project, discovered by reflection, and executed by name.

Recommended workflow

  1. Record the real user session if the bug is difficult to reproduce by hand.
  2. Replay the recording through InkkOops so the app can capture a final frame and telemetry automatically.
  3. Turn the replay into a focused built-in repro script if the bug needs repeated iteration.
  4. Add or update a regression test in InkkSlinger.Tests once the root cause is understood.
