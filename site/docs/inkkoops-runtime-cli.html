InkkOops

Runtime entry points and CLI usage

InkkOops can be started directly through the app command line or through the companion CLI project. Both paths target the same in-process runtime service.

App command line

The app accepts these InkkOops flags:

  • --inkkoops-script <name>: run a built-in script at startup.
  • --inkkoops-recording <path>: replay a previously recorded session.
  • --inkkoops-record: record user interaction and write it on normal app shutdown.
  • --inkkoops-artifacts <path>: override the artifact output root for script and replay runs.
  • --inkkoops-record-root <path>: override the root directory for recorded sessions.
  • --inkkoops-pipe <name>: choose the named pipe used for attach mode.
  • --inkkoops-disable-retained: diagnostic mode for turning retained rendering off.
  • --inkkoops-disable-dirty: diagnostic mode for turning dirty-region rendering off.
dotnet run --project InkkSlinger.csproj -- --inkkoops-script "datagrid-workbench-capture"

dotnet run --project InkkSlinger.csproj -- --inkkoops-script "buttons-resize-chaos-repro" --inkkoops-artifacts "artifacts/my-chaos-run"

dotnet run --project InkkSlinger.csproj -- --inkkoops-recording "C:\Users\Shadow\Desktop\InkkSlinger\artifacts\inkkoops-recordings\20260328-001157740-recorded-session\recording.json"

Companion CLI

The CLI wraps common runtime operations so you do not have to remember the app flags by hand.

dotnet run --project InkkOops.Cli/InkkOops.Cli.csproj -- list

dotnet run --project InkkOops.Cli/InkkOops.Cli.csproj -- run --script "sidebar-button-richtextbox" --launch

dotnet run --project InkkOops.Cli/InkkOops.Cli.csproj -- run --script "sidebar-button-richtextbox" --attach

dotnet run --project InkkOops.Cli/InkkOops.Cli.csproj -- record --launch

Launch mode vs attach mode

--launch

Starts a new app process with a startup request. Use this when the scenario should begin from a cold start or when no app instance is already running.

--attach

Connects to the app's named pipe and submits a script request to the already-running runtime service. Use this when you want to keep the app open and inject work into it.

Attach mode is serialized. If a script is already queued or running, the runtime returns a busy response rather than running multiple scripts concurrently.

Named pipe and artifact overrides

dotnet run --project InkkOops.Cli/InkkOops.Cli.csproj -- run `
  --script "buttons-resize-hover-repro" `
  --attach `
  --pipe "inkkoops-dev" `
  --timeout 45000 `
  --artifacts "artifacts/inkkoops-dev"

Use a dedicated pipe name if you need multiple local development sessions that should not talk to each other.

Automatic exit for startup runs

Startup script runs and startup recording replays are treated as single-purpose sessions. The runtime executes the request, writes artifacts, and then requests app exit automatically.

Recorded user sessions still finalize on normal shutdown. If you are capturing a recording, close the app normally rather than force-killing the process.

Diagnostic switches

--inkkoops-disable-retained and --inkkoops-disable-dirty are there to isolate renderer classes of bugs. They are not normal operating modes.

xUnit example: launch the Controls Catalog from a test

The app boots into the Controls Catalog by default. A host-side xUnit test can launch the app as a separate process, run a built-in InkkOops script against that live runtime, wait for the process to exit, and then assert on the generated artifact bundle.

using System.Diagnostics;
using System.IO;
using System.Linq;
using System.Text.Json;
using Xunit;

public sealed class ControlsCatalogRuntimeTests
{
    [Fact]
    public void ControlsCatalog_CanLaunch_And_Run_A_BuiltIn_Script()
    {
        var repoRoot = @"C:\Users\Shadow\Desktop\InkkSlinger";
        var artifactRoot = Path.Combine(repoRoot, "artifacts", "inkkoops-xunit");

        if (Directory.Exists(artifactRoot))
        {
            Directory.Delete(artifactRoot, recursive: true);
        }

        var startInfo = new ProcessStartInfo
        {
            FileName = "dotnet",
            WorkingDirectory = repoRoot,
            UseShellExecute = false
        };

        startInfo.ArgumentList.Add("run");
        startInfo.ArgumentList.Add("--project");
        startInfo.ArgumentList.Add("InkkSlinger.csproj");
        startInfo.ArgumentList.Add("--");
        startInfo.ArgumentList.Add("--inkkoops-script");
        startInfo.ArgumentList.Add("sidebar-button-richtextbox");
        startInfo.ArgumentList.Add("--inkkoops-artifacts");
        startInfo.ArgumentList.Add(artifactRoot);

        using var process = Process.Start(startInfo)!;
        Assert.True(process.WaitForExit(120_000), "The app did not exit within the timeout.");
        Assert.Equal(0, process.ExitCode);

        var runDirectory = Directory.GetDirectories(artifactRoot).Single();
        var resultPath = Path.Combine(runDirectory, "result.json");
        Assert.True(File.Exists(resultPath), "The runtime did not write result.json.");

        using var document = JsonDocument.Parse(File.ReadAllText(resultPath));
        Assert.Equal(
            "Completed",
            document.RootElement.GetProperty("status").GetString());
    }
}

This is a host-side integration test. It is not using InkkOopsTestHost. It is starting the real app process, letting the startup script drive the live Controls Catalog, and then asserting on the runtime artifacts after the app exits.

xUnit example: replay a recording from a test

using System.Diagnostics;
using System.IO;
using System.Linq;
using System.Text.Json;
using Xunit;

public sealed class ControlsCatalogReplayTests
{
    [Fact]
    public void Recorded_Session_Can_Be_Replayed()
    {
        var repoRoot = @"C:\Users\Shadow\Desktop\InkkSlinger";
        var artifactRoot = Path.Combine(repoRoot, "artifacts", "inkkoops-replay-xunit");
        var recordingPath = Path.Combine(
            repoRoot,
            "artifacts",
            "inkkoops-recordings",
            "20260328-120250760-recorded-session",
            "recording.json");

        var startInfo = new ProcessStartInfo
        {
            FileName = "dotnet",
            WorkingDirectory = repoRoot,
            UseShellExecute = false
        };

        startInfo.ArgumentList.Add("run");
        startInfo.ArgumentList.Add("--project");
        startInfo.ArgumentList.Add("InkkSlinger.csproj");
        startInfo.ArgumentList.Add("--");
        startInfo.ArgumentList.Add("--inkkoops-recording");
        startInfo.ArgumentList.Add(recordingPath);
        startInfo.ArgumentList.Add("--inkkoops-artifacts");
        startInfo.ArgumentList.Add(artifactRoot);

        using var process = Process.Start(startInfo)!;
        Assert.True(process.WaitForExit(120_000), "The replay run did not exit within the timeout.");
        Assert.Equal(0, process.ExitCode);

        var runDirectory = Directory.GetDirectories(artifactRoot).Single();
        Assert.True(File.Exists(Path.Combine(runDirectory, "result.json")));
        Assert.True(Directory.GetFiles(runDirectory, "*.png").Length > 0);
        Assert.True(Directory.GetFiles(runDirectory, "*.txt").Length > 0);

        using var document = JsonDocument.Parse(
            File.ReadAllText(Path.Combine(runDirectory, "result.json")));
        Assert.Equal(
            "Completed",
            document.RootElement.GetProperty("status").GetString());
    }
}

Use this pattern when the bug is driven by a human recording and you want xUnit to execute the exact same runtime replay that you use manually from the shell.
