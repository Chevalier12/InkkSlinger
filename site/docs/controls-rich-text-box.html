RichTextBox

RichTextBox is the structured document editor for rich text, inline formatting, hyperlinks, lists, tables, clipboard round-trips, and undoable editing. It hosts a FlowDocument, renders the document surface directly, maintains caret and selection state, and exposes routed events for document changes and hyperlink navigation.

Quick start

Use RichTextBox when you need editable document content rather than a single plain text buffer. The control works directly against a FlowDocument, so paragraphs, runs, lists, tables, and hyperlinks remain part of the editing model.

Minimal editor with a document

<RichTextBox Width="420"
             Height="220"
             Background="{StaticResource DarkSurfaceBrush}"
             BorderBrush="{StaticResource DarkBorderBrush}"
             BorderThickness="1"
             Padding="8" />

Seed a FlowDocument in code

var document = new FlowDocument();
var paragraph = new Paragraph();
paragraph.Inlines.Add(new Run("Release notes"));
document.Blocks.Add(paragraph);

editor.Document = document;

Read-only viewer with rich selection

<RichTextBox Width="520"
             Height="260"
             IsReadOnly="True"
             TextWrapping="Wrap" />

Default model: the editor always keeps a valid FlowDocument. Setting Document to null is coerced back to a default document with one empty paragraph.

Document model

RichTextBox edits a FlowDocument, not a plain string property. The document can contain block-level and inline rich content, and the editor preserves that structure through many selection, clipboard, and command operations.

Block content

The document can host Paragraph, List, Table, Section, and related flow blocks as part of the editable surface.

Inline content

Paragraphs can contain Run, Bold, Italic, Underline, Hyperlink, and other inline spans. Formatting commands operate against that structure instead of flattening it to plain text.

  • DocumentChanged is raised both when you replace Document and when the current document mutates internally.
  • The control applies implicit hyperlink styling when documents change so links participate in the visual model without requiring manual post-processing.
  • The editor clamps caret and selection offsets whenever document structure changes, which keeps document edits and undo or redo transitions stable.

Document with inline rich content

var document = new FlowDocument();
var paragraph = new Paragraph();

paragraph.Inlines.Add(new Run("Build status: "));

var bold = new Bold();
bold.Inlines.Add(new Run("Ready"));
paragraph.Inlines.Add(bold);

var link = new Hyperlink
{
    NavigateUri = "https://example.com/release-notes"
};
link.Inlines.Add(new Run(" release notes"));
paragraph.Inlines.Add(link);

document.Blocks.Add(paragraph);
editor.Document = document;

Caret, selection, and pointer behavior

The editor tracks caret and selection directly through text offsets into the flattened document text. Public inspection surfaces include CaretIndex, SelectionStart, and SelectionLength, while input handlers and command bindings keep those values synchronized with the document layout.

Keyboard navigation

Arrow keys move by character or visual line. Ctrl+Left and Ctrl+Right move by word, Home and End move by line, and Ctrl+Home or Ctrl+End jump to document bounds.

Pointer selection

Pointer down moves the caret or extends selection, drag updates the active selection range, double-click selects a word, and triple-click selects a paragraph.

  • Shift extends selection during keyboard navigation.
  • The editor auto-scrolls while pointer selection drags beyond the visible text rect.
  • The caret visibility is maintained automatically, including horizontal offset updates for long unwrapped lines.
  • Read-only mode still allows caret movement, selection, and hyperlink activation.

Inspect selection state

Logger.Write($"Caret: {editor.CaretIndex}");
Logger.Write($"Selection start: {editor.SelectionStart}");
Logger.Write($"Selection length: {editor.SelectionLength}");

Programmatic focus handoff from input hosting

editor.SetFocusedFromInput(true);
editor.SetMouseOverFromInput(true);

Editing commands and shortcuts

RichTextBox registers a broad editing command surface through command bindings and key bindings. Most text editing, selection, formatting, clipboard, list, and table operations route through those commands, which keeps keyboard shortcuts and explicit CommandManager.Execute calls aligned.

Core editing

Backspace, delete, paragraph break, line break, tab, shift+tab, copy, cut, paste, undo, redo, and select-all are all part of the active editing path.

Rich editing

Ctrl+B, Ctrl+I, and Ctrl+U toggle bold, italic, and underline. Additional editing commands support list indent or outdent plus table insertion, cell split, and cell merge.

  • Ctrl+Z and Ctrl+Y drive undo and redo directly through the internal document undo manager.
  • Typing bursts are grouped so sequential typing can undo as a single operation.
  • When no selection exists, bold, italic, and underline commands switch typing mode so newly typed text inherits the chosen inline style.
  • When a selection exists, formatting commands apply or remove inline wrappers across the selected rich fragment.

Execute a formatting command explicitly

CommandManager.Execute(EditingCommands.ToggleBold, null, editor);
CommandManager.Execute(EditingCommands.ToggleItalic, null, editor);
CommandManager.Execute(EditingCommands.ToggleUnderline, null, editor);

Execute structural commands

CommandManager.Execute(EditingCommands.InsertTable, null, editor);
CommandManager.Execute(EditingCommands.IncreaseListLevel, null, editor);
CommandManager.Execute(EditingCommands.DecreaseListLevel, null, editor);

Clipboard, lists, and tables

The control supports rich clipboard and structure-aware editing paths rather than treating everything as raw plain text. This matters most when selections cross paragraph boundaries or when the document already contains rich inline or block content.

Clipboard

Copy and paste handle both plain text and richer document fragments. The editor recognizes rich clipboard payloads such as RTF, XAML, and XamlPackage when available.

Structure-aware edits

List and table commands preserve document structure where possible, including nested list operations, table insertion, and adjacent cell split or merge behavior.

  • Enter inserts a paragraph break, while Shift+Enter inserts a line break.
  • Tab and Shift+Tab participate in list-level and table-cell editing instead of immediately handing focus away.
  • Table insertion creates a default 2x2 table and moves the caret into the first reachable cell.
  • Merge and split commands act on the currently active table cell and are gated by the current document position.

Insert a table and then edit cells

CommandManager.Execute(EditingCommands.InsertTable, null, editor);
CommandManager.Execute(EditingCommands.SplitCell, null, editor);
CommandManager.Execute(EditingCommands.MergeCells, null, editor);

Hyperlinks and read-only behavior

RichTextBox supports interactive hyperlinks inside the document model. Hyperlinks are resolved from the current caret or pointer hit-test offset and raise a bubbled HyperlinkNavigate routed event carrying the target URI.

Keyboard activation

Ctrl+Enter activates the hyperlink at the current selection or caret position. This remains available even in read-only mode.

Pointer activation

In read-only mode, a simple click on a hyperlink can activate it when the pointer interaction did not turn into a drag selection.

  • Read-only blocks text mutation, cut, paste, and formatting changes.
  • Read-only still permits navigation, selection, mouse-over tracking, and hyperlink activation.
  • The control maintains hovered hyperlink state separately from the selected text range.

Handle navigation events

editor.HyperlinkNavigate += (_, args) =>
{
    Logger.Write($"Navigate: {args.NavigateUri}");
};

Layout, rendering, and scrolling

The control renders its own document surface. It measures document content through the layout engine, draws the background and border, paints selection geometry and inline runs, draws table borders, and finally renders the caret when focused and visible.

Measurement

Desired size comes from the laid out document plus Padding and BorderThickness. With TextWrapping="NoWrap", measurement uses infinite text width.

Scrolling

The editor manages internal horizontal and vertical offsets itself. Mouse wheel scrolling is vertical and requires focus, while caret visibility logic can also adjust horizontal offsets for long lines.

  • The control is not driven by a PART_ContentHost or ScrollViewer template contract.
  • Selection, caret, and rendered runs are clipped to the computed text rect inside the control's border and padding.
  • When focused, the caret blinks on a half-second cadence and the editor participates in the UI root frame update loop.
  • Pointer selection auto-scroll can move both vertically and horizontally when dragging beyond the visible document bounds.

Configure no-wrap editing

<RichTextBox Width="500"
             Height="180"
             TextWrapping="NoWrap"
             Padding="8"
             BorderThickness="1" />

Performance and diagnostics

The editor carries internal performance tracking for layout, rendering, selection geometry, clipboard serialization and deserialization, and edit operations. This is useful when profiling large documents or validating undo and clipboard costs during complex flows.

Snapshot access

GetPerformanceSnapshot() returns the current metrics summary, including undo stack visibility through the integrated performance tracker.

Reset support

ResetPerformanceSnapshot() clears the tracked counters so a new scenario can be measured without constructing a new editor instance.

Capture a performance snapshot

var snapshot = editor.GetPerformanceSnapshot();
Logger.Write(snapshot.ToString());

editor.ResetPerformanceSnapshot();

Theme and styling

The current theme styles RichTextBox through direct property setters in App.xml. Unlike TextBox and PasswordBox, the theme does not replace the visual tree with a control template because the editor already renders its own document surface.

Theme-set properties

Background, Foreground, CaretBrush, SelectionBrush, BorderBrush, BorderThickness, and Padding are set from the default dark theme.

Control-owned rendering

Those values feed directly into the control's renderer instead of being forwarded to named template parts.

Current theme entry

<Style TargetType="{x:Type RichTextBox}">
  <Setter Property="Background" Value="{StaticResource DarkSurfaceBrush}" />
  <Setter Property="Foreground" Value="{StaticResource TextPrimaryBrush}" />
  <Setter Property="CaretBrush" Value="{StaticResource OrangePrimaryBrush}" />
  <Setter Property="SelectionBrush" Value="{StaticResource OrangeSubtleBrush}" />
  <Setter Property="BorderBrush" Value="{StaticResource DarkBorderBrush}" />
  <Setter Property="BorderThickness" Value="1" />
  <Setter Property="Padding" Value="8" />
</Style>

Property and event reference

Core document API

Document: active FlowDocument being edited or viewed.

DocumentChanged: bubbled routed event raised when the active document is replaced or mutated.

HyperlinkNavigate: bubbled routed event raised when a hyperlink is activated.

Editing state

CaretIndex: current caret offset into the flattened document text.

SelectionStart and SelectionLength: current selection bounds.

IsReadOnly: blocks mutation while preserving navigation and selection.

Visual properties

Background, Foreground, BorderBrush, and BorderThickness: direct rendering inputs for the editor chrome.

Padding: inset around the document text rect.

SelectionBrush and CaretBrush: colors for selection geometry and caret rendering.

TextWrapping: wrap or no-wrap layout behavior.

Diagnostics

GetPerformanceSnapshot(): capture performance counters for layout, render, clipboard, and edit activity.

ResetPerformanceSnapshot(): clear the current counters.

IsFocused and IsMouseOver: input-driven visual state surfaces.

Patterns and examples

Release-notes editor

<RichTextBox Width="560"
             Height="320"
             Padding="12"
             SelectionBrush="{StaticResource OrangeSubtleBrush}"
             CaretBrush="{StaticResource OrangePrimaryBrush}" />

React to document changes

editor.DocumentChanged += (_, _) =>
{
    Logger.Write("Document changed");
};

Read-only rich viewer with active links

<RichTextBox Width="520"
             Height="260"
             IsReadOnly="True"
             TextWrapping="Wrap" />

Notes and pitfalls

Document, not Text. This control does not expose a simple Text dependency property. If you need plain-text editing only, TextBox is the simpler surface.

No TextBox-style content host. The editor renders its own document, selection, caret, and table borders. There is no PART_ContentHost scroll host contract to preserve in a template.

Read-only still allows interaction. IsReadOnly blocks mutation, but navigation, selection, wheel scrolling, and hyperlink activation remain part of the runtime behavior.

Structural edits can stay rich. Clipboard, formatting, list, and table commands often preserve existing document structure instead of flattening content to plain text, which is usually what you want but is different from simple text replacement semantics.
